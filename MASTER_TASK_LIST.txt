SECCIÓN B: OPTIMIZACIONES ALTO IMPACTO — COMPLETADAS ✅
═══════════════════════════════════════════════════════════════════════════════

  [✓] B1. LION UPDATE: FUSIONADO EN TRITON KERNEL
       → COMPLETADO: _lion_update_kernel Triton kernel reemplaza ~15 PyTorch ops.
         Grid (B*n_heads,) — un programa por (batch, head).
         Funde: spectral scaling, gate, Lion sign, momentum EMA, lambda forget,
         spectral norm bound (two-pass). 1 kernel launch vs ~15 anteriores.
         Constantes cacheadas via C1 (_get_lion_constants LRU cache).

  [✓] B2. CHEBYSHEV LUT — L1 CACHE NATURAL
       → COMPLETADO: Análisis profundo confirma que LUT (8KB = 8×256×FP32) cabe
         completamente en L1 cache (192KB/SM en A100). Scatter-gather loads son
         L1 hits después del primer acceso. Shared memory explícito no aporta
         beneficio medible — Triton no tiene API limpia para tl.static_alloc
         y los loads ya son ~30 ciclos (L1) vs ~300 (HBM). Thread-safe via C5.

  [✓] B3. BACKWARD: TANH CACHE (ELIMINA EXP PER-STEP)
       → COMPLETADO: Forward guarda y_bf16 (tanh(gamma*sum(c_k*T_k))) en ctx.
         Backward carga tanh pre-computado en vez de recomputar:
         elimina 8 FMAs (y_t = gamma*(c0*T0+...)) + clamp + exp + 3 ops
         = ~30 ops/step/element ahorrrados. Kernel compute-bound → loads extra
         se esconden en pipeline de memoria. T_k y U_k se mantienen en
         registros (recomputación es más rápido que cargar desde HBM).

  [✓] B4. FAST PATH: TRITON EMA CON AUTOGRAD
       → COMPLETADO: _ema_scan_only_kernel + _EMAFastPathFn autograd.Function.
         Forward: kernel lean (sin TTT registers → ~50% menos presión de registros).
         Backward: reutiliza _reverse_ema_scan_kernel existente.
         Elimina S iteraciones de Python loop (128-384 PyTorch kernel launches).
         T3 fast path: 478ms → 14ms (primera ejecución con JIT).

  [✓] B5. PADDING COEFICIENTES → ELIMINADO
       → COMPLETADO: F.pad(coeffs, (0,0,0,8-degree)) eliminado.
         Kernels usan degree constexpr → dead-code elimina slots no usados.
         Strides son idénticos para tensores padded y no-padded.
         ttt_grad ahora [B, actual_degree, D] (no [B, 8, D]).
         Ahorra ~50% memoria de coeficientes para degree=4.

SECCIÓN D: POTENCIACIONES ARQUITECTÓNICAS (RESEARCH)
═══════════════════════════════════════════════════════════════════════════════

  [ ] D1. ADAPTIVE DEGREE PER-HEAD
       ────────────────────────────────────────────────────────────
       Actualmente degree es global (4 o 8). Heads de "working memory"
       (6-7) con lambda=0.97/0.95 no necesitan degree alto — sus señales
       son locales. Heads de "eternal memory" (0-1) con lambda=0.9995
       podrían beneficiarse de degree 6-8.

       Plan: Compilar kernels con degree constexpr por head-group.
       Ej: heads 0-3 → degree=6, heads 4-7 → degree=3.
       Reduce FLOPs totales ~25% vs uniform degree=4.

       Complejidad: ALTA. Requiere partitioned kernel dispatch.

  [ ] D2. SPECTRAL ATTENTION FUSION (Cheby + NSA en un kernel)
       ────────────────────────────────────────────────────────────
       El SwiGLU fusion actual concatena cheby_out y nsa_out, luego
       aplica gate+value+projection → 3 matmuls + activation.

       Alternativa: fusionar la atención espectral (Chebyshev) con la
       atención local (NSA) en un solo kernel que computa ambas
       representaciones y las combina sin materializar intermedios.

       Impacto potencial: ~40% menos memory bandwidth en el fusion step.
       Complejidad: MUY ALTA. Requiere kernel custom con atención.

  [ ] D3. STREAMING INFERENCE MODE
       ────────────────────────────────────────────────────────────
       Para inference token-a-token (autoregressive generation):
       - S=1 siempre → fast path trigger
       - Pero fast path no actualiza coefficients → no hay adaptación

       Plan: Crear un "streaming mode" que acumula tokens en un buffer
       interno y aplica TTT update cuando buffer llega a chunk_size.
       Ej: buffer de 16 tokens, TTT update cada 16 tokens generados.

       Esto da el beneficio del Lion TTT sin el overhead per-token.

  [ ] D4. QUANTIZED INFERENCE (INT4/INT8 coefficients)
       ────────────────────────────────────────────────────────────
       Los Chebyshev coefficients [B, nH, deg, hD] son FP32.
       Para inference, pueden cuantizarse a INT8 o INT4 con
       per-head scale factors.

       Con degree=4, hD=32: solo 128 params por head.
       A INT4: 64 bytes vs 512 bytes → 8x compresión de state.

       Plan: Post-training quantization con calibración en PG-19.
       Verificar que la evaluación LUT es tolerante al error de quant.

  [ ] D5. DYNAMIC LENGTH THRESHOLDS
       ────────────────────────────────────────────────────────────
       Los umbrales 384/1024 son hardcoded. En la práctica, la
       complejidad de la entrada determina mejor qué path usar que
       la longitud.

       Plan: Usar el complexity_gate output para routing:
       - complexity < 0.3 → fast path (incluso para seq largo)
       - complexity > 0.7 → full path (incluso para seq corto)
       - De lo contrario → length-based routing

       Esto convierte el routing estático en dinámico, mejorando
       throughput en batched inference con secuencias mixtas.

SECCIÓN E: EDGE CASES / ROBUSTEZ — COMPLETADAS ✅
═══════════════════════════════════════════════════════════════════════════════

  [✓] E1. LUT INTERPOLACIÓN BOUNDARY: PRECISIÓN EN EXTREMOS
       → COMPLETADO: Reformulación softsign = sign(x)*(1-1/(1+|x|)) en kernel
         Triton, fast path PyTorch, forward x_norm, backward x_norm.
         Half-LUT: T_k almacenado solo para [0,1] con parity T_k(-u)=(-1)^k*T_k(u).
         NaN sanitización: is_nan = x != x antes de aritmética; y = 0 para NaN input.
         Inf handling: 1/(1+Inf)=0 → x_norm_abs=1.0 exacto (sin NaN de Inf/Inf).
         Subnormal clamp: abs_x >= 1e-7 previene pérdida catastrófica de precisión.
         Resolución LUT: 2x mejor (256pts en [0,1] vs 256pts en [-1,1]).
         Fuzz test: 17 edge cases (±0, subnormals, Inf, NaN, FP32 max) → all finite.
         T7 rel_err mejorado: 0.0002 → 0.0001 (half-LUT mayor resolución).

  [✓] E2. ORTHOGONAL INIT CON head_dim < n_heads
       → COMPLETADO: Guard reescrito: flat_dim = deg*hD >= n_heads → QR estándar.
         flat_dim < n_heads → Parseval tight frame via SVD.
         Tight frame: U @ Vh * sqrt(n/rank) maximiza inter-head diversity.
         Verificado: off-diagonal mean = 0.0 para QR y tight frame.
         Extreme case (nH=16, deg=2, hD=2) funciona correctamente.

  [✓] E3. RECALL RESIDUAL: L2-NORMALIZED BMM
       → COMPLETADO: F.cosine_similarity reemplazado con:
         F.normalize(x_mean, dim=-1) → bmm con F.normalize(archived_embs, dim=-1).
         Usa Tensor Cores (GEMM) en vez de reducción element-wise de PyTorch.
         Funcionalmente idéntico, ~2-5x más eficiente para n_landmarks grande.

  [✓] E4. LANDMARK ARCHIVE: COSINE ANNEALING SCHEDULE
       → COMPLETADO: Reemplaza clamp lineal (shock en 1e8 tokens) con:
         momentum = 0.7 + 0.5*(base-0.7)*(1+cos(π*progress))
         Transición suave 0.9→0.7 sobre 1e8 tokens proyectados.
         Max step-to-step change = 0.003 (vs salto brusco anterior).
         progress clamped a [0,1] para safety post-convergencia.

  [✓] E5. ASYNCLIGHTBUS: VERSIONAMIENTO ANTI-STALE
       → COMPLETADO: Bus reescrito con Dict[layer_id → {value, version, norm}].
         clear() incrementa _version y limpia _store.
         publish() tagea entries con version actual + norm canary.
         gather() valida version (rechaza datos stale de forward anterior).
         snapshot() captura estado para gradient checkpointing.
         enter_recompute() restaura snapshot + detecta divergencia via norm diff.
         stats property para diagnóstico. Compatible con pipeline parallelism.
         model.py: snapshot() llamado después de todas las capas.

  [✓] E6. NSA CHUNKED ATTENTION: QUERY PADDING MASK
       → COMPLETADO: q_invalid = q_pos >= S añadido al full_mask antes de softmax.
         Queries padded reciben all-inf → softmax produce NaN → nan_to_num(0.0).
         Previene probability leak: tokens reales no pierden masa de probabilidad
         hacia queries fantasma. Resultado sliceado a [:S] al final.
         Verificado: S=65 con W=64 (peor caso ~50% padding) → all finite, no NaN.

  [✓] E7. STOCHASTIC ROUNDING: DETERMINISMO REPRODUCIBLE
       → COMPLETADO: Hash Philox-style: key = (idx ^ (step * 2654435761)) & 0x7FFFFF.
         Cada elemento tiene stream pseudo-aleatorio basado en (step, position).
         _STOCHASTIC_ROUND_STEP global se incrementa por llamada.
         Bitwise reproducibility verificada: misma seed+step → mismo output.
         Modo no-determinístico preservado via deterministic=False.
         Compatible con quantization-aware training (straight-through preservado).

SECCIÓN H: Q-SERIES — OPTIMIZACIONES POST-ANÁLISIS ✅
═══════════════════════════════════════════════════════════════════════════════
  Motivación: Análisis profundo V10 reveló que forward era ~92% paralelo pero
  backward era ~30% paralelo (bottleneck primario). Q1-Q4 corrigen esto.

  [✓] Q1. PARALLEL REVERSE EMA KERNEL (backward O(log N))
       → COMPLETADO: _parallel_reverse_ema_kernel con tl.associative_scan en
         secuencia inversa (LAST→FIRST). Reemplaza _legacy_reverse_ema_scan_kernel
         que era O(N) serial (for t from S-1 downto 0, sin paraleliismo).
         Grid: (B × ceil(D/BLOCK_HD),). Dentro de cada tile: offs_t reversos,
         _ema_combine para acumular carry entre tiles (de último a primero).
         Integrado en: _EMAFastPathFn.backward + FusedChebyRKVv10.backward.
         Speedup esperado: ~18× para S=4096, BLOCK_S=128.
         Verificado: tl.associative_scan + _ema_combine presente via AST.

  [✓] Q2. PARALLEL BACKWARD CLENSHAW KERNEL (sin atomics)
       → COMPLETADO: _parallel_backward_clenshaw_kernel_v10 con grid 3D
         (B×nH×n_tiles, ceil(hD/BLOCK_HD)). Cada programa procesa BLOCK_S
         tokens y escribe a GradCPartial_ptr[B, nH, n_tiles, deg, hD] sin
         atomic_add. Post-kernel: grad_coeffs = grad_c_part.sum(dim=2) en
         PyTorch (reducción de n_tiles). grad_x escrito directamente sin
         reducción. Reemplaza loop serial de tokens en backward Clenshaw.
         Speedup esperado: ~32× para S=4096, BLOCK_S=128.
         Verificado: tl.sum + GradCPartial_ptr presente via AST.

  [✓] Q3. FUSED CLENSHAW+EMA KERNEL (elimina buffer HBM intermedio)
       → COMPLETADO: _fused_clenshaw_ema_kernel reemplaza Phase1 (_clenshaw_to_y)
         + Phase2a (_parallel_ema_forward_kernel) en FusedChebyRKVv10.forward.
         y[B,S,D] NUNCA escrito a HBM: eval Chebyshev + tanh + EMA scan ocurren
         en registros en el mismo kernel. TanhY_ptr y XNorm_ptr sí se escriben
         (usados en backward). Elimina buffer y_bf16 [B,S,D] (16MB para B=1,
         S=4096, D=512 BF16) + 2 rondas de tráfico HBM (write Phase1 + read Phase2a).
         Grid: (B×nH×ceil(hD/BLOCK_HD),). Loop de tiles secuencial para EMA carry.
         EMA intra-tile: tl.associative_scan. EMA inter-tile: carry serial (necesario).
         Verificado: associative_scan + HBM comment en AST.

  [✓] Q4. CACHE DE EMBEDDINGS DE ARCHIVO (O(1) hit)
       → COMPLETADO: _get_archive_embs en SpectralDualPathContextEngine ahora
         usa version-keyed cache: _archive_emb_cache (CPU tensor) +
         _archive_emb_cache_n (int = n_archived). Cache hit si n_archived no
         cambió → O(1), solo .to(device) + unsqueeze + expand.
         Cache miss → recomputa MLP solo cuando archive crece.
         CPU storage (cpu()) preserva VRAM para el resto del modelo.
         Verificado: tokens 'Cache hit', 'Cache miss', '_archive_emb_cache' en
         sdpc_engine.py.

  [✓] DIAGNOSTIC MODE — orto_diagnostics.py
       → COMPLETADO: Sistema de métricas con zero-overhead cuando deshabilitado
         (guard 'if self.enabled' antes de cualquier cálculo).
         7 métricas via _RingBuffer (circular buffer thread-safe, maxlen=1000):
