§2. VERSIÓN 1 — ARQUITECTURA BASE
================================================================================

2.1 KERNEL TRITON (sdpc_kernel.py)
  - Evaluación de Clenshaw fusionada en kernel Triton
  - Recurrencia de grado 4: T₀, T₁, T₂, T₃
  - Domain Throttling: Softsign x/(1+|x|) para mapear a [-1,1]
  - EMA State: s_t = μ·s_{t-1} + (1-μ)·y_t (μ=0.9)
  - TTT Delta Rule: c_k ← c_k + η·err·T_k(x̂) (η=0.01)
  - Hard Clamping: c_k ∈ [-1, 1] para prevenir "resonance catastrophe"
  - Grid: (batch, cdiv(dim, BLOCK_D))

2.2 NSA MODULE (nsa_module.py)
  - Multi-Head Attention con PyTorch SDPA (FlashAttention backend)
  - Causal sliding window + Landmark cross-attention
  - QKV projection unificada para throughput

2.3 SDPC ENGINE (sdpc_engine.py)
  - Dual-Path: Cheby-RKV (global) + NSA (local)
  - RMSNorm para estabilidad
  - SwiGLU Gated Fusion: SiLU(gate) ⊙ value
  - Dynamic Degree Adaptation: K=2 si complejidad < 0.3, K=4 si no
  - Periodic Soft Reset: state *= 0.5 cada 65,536 tokens

2.4 RESULTADOS V1
  - 65K tokens: 1.9s forward, 3.9s backward
  - VRAM: ~2.8 GB para 65K tokens
  - Estabilidad: PASS (sin NaN/Inf)
  - Copy Task (100 steps): Loss 256 → 129 (50% mejora)

§4. VERSIÓN 2 — 5 MEJORAS ARQUITECTÓNICAS
================================================================================

4.1 MEJORA 1: CHUNK-PARALLEL PROCESSING (sdpc_kernel_v2.py)
  Problema: Bucle `for s in range(seq_len)` era completamente secuencial
  Solución: Procesamiento por chunks de CHUNK_SIZE=256 tokens
  - Dentro de cada chunk: Clenshaw eval con coeficientes FROZEN
  - TTT gradients se acumulan sobre el chunk completo
  - Update de coeficientes UNA VEZ por chunk (no por token)
  - Esto permite vectorización parcial del Clenshaw dentro del chunk
  Resultado: Throughput comparable, con mejor paralelización GPU

4.2 MEJORA 2: ADAM-LITE TTT (dentro del kernel Triton)
  Problema: TTT con delta rule básica (c += lr * err * T_k) es ruidosa
  Solución: Optimizer Adam completo dentro del kernel Triton
  - 1st moment (momentum): m1 = β₁·m1 + (1-β₁)·g (β₁=0.9)
  - 2nd moment (variance): m2 = β₂·m2 + (1-β₂)·g² (β₂=0.999)
  - Update: c += lr · m1 / (√m2 + ε)
  - Los moments se persisten entre chunks (almacenados en VRAM)
  Resultado: Convergencia más suave, menos ruido en updates

4.3 MEJORA 3: PROYECCIÓN NO-LINEAL SiLU
  Problema: Salida lineal de Clenshaw limita representaciones
  Solución: Aplicar SiLU(y) = y·σ(y) post-Clenshaw dentro del kernel
  - σ(y) = 1/(1+exp(-y)) computado en registros GPU
  - Zero overhead (fusionado en el mismo kernel)
  Resultado: Estado almacena relaciones no-lineales más ricas

4.4 MEJORA 4: RMSNorm SOBRE COEFICIENTES (reemplaza clamping)
  Problema: Hard clamping c_k ∈ [-1,1] causa "dead zones" (saturation)
  - Coeficientes pegados en ±1 dejan de reaccionar a nueva información
  Solución: RMSNorm soft bounding
  - RMS = √(mean(c_k²))
  - Si RMS > 1.0: c_k = c_k / RMS (normalización unitaria)
  - Si RMS ≤ 1.0: no se toca (sin restricción innecesaria)
  - Preserva la DIRECCIÓN del vector, solo acota la ENERGÍA
  Resultado: Max per-head RMS = 0.14, sin puntos muertos

4.5 MEJORA 5: MULTI-HEAD CHEBYSHEV STATE
  Problema: Un solo estado global mezcla todas las frecuencias
  Solución: n_cheby_heads cabezas independientes
  - Cada cabeza tiene: coeficientes [degree × head_dim], m1, m2
  - Learning rate variable por cabeza: lr_h = base_lr · (1 + 0.2·(h - H/2)/H)
  - Cabezas bajas → lr menor (memoria larga, estable)
  - Cabezas altas → lr mayor (reactivas, corto plazo)
  Resultado: coeffs [B, 4, 4, 64] — 4 cabezas espectrales independientes

4.6 COMPATIBILIDAD V1
  - apply_cheby_rkv() sigue funcionando vía wrapper transparente
  - Convierte estado V1 [B, deg, D] → V2 [B, 1, deg, D] internamente

4.7 RESULTADOS V2

§5. VERSIÓN 3 — LANDMARK ARCHIVING + 10M TOKENS
================================================================================

5.1 PROBLEMA: OLVIDO PROGRESIVO POR DILUCIÓN
  Con EMA momentum=0.9, el peso del token #N decrece como 0.9^(current-N).
  Para token #1 en posición 1,000,000: peso = 0.9^999999 ≈ 10^(-45,757)
  → Información del token #1 se pierde completamente.
  
  La RMSNorm de V2 previene la saturación, pero NO previene la dilución.
  Se necesita un mecanismo de "memoria perfecta" externo al EMA.

5.2 SOLUCIÓN: LANDMARK STATE ARCHIVING (landmark_archive.py)

  Concepto: Congelar snapshots del estado de Chebyshev periódicamente y
  permitir que la atención "mire atrás" a esos estados congelados.

  Implementación:
  
  a) LandmarkArchive — Buffer circular con merge jerárquico
     - Cada archive_interval tokens (ej: 131,072), se congela el estado actual
     - Almacena: [n_cheby_heads, degree, head_dim] por landmark
     - Costo: 4 KB por landmark (para d=256, degree=4)
     - max_landmarks: 32 → 128 KB total máximo
     
  b) Hierarchical Merge — Compresión cuando el buffer está lleno
     - Cuando n_landmarks == max_landmarks:
     - Los pares más antiguos se promedian: (L0+L1)/2, (L2+L3)/2, ...
     - Libera espacio para nuevos landmarks recientes
     - Los landmarks recientes NUNCA se tocan (son los más valiosos)
     - Resultado: O(log(T)) landmarks para T tokens procesados
     
  c) State-to-Embedding Projection
     - MLP: [n_heads × degree × head_dim] → [d_model]
     - Convierte estados de Chebyshev congelados en embeddings
     - Los embeddings son compatibles con la atención del NSA
     
  d) Global Summary Landmark
     - Weighted average de todos los landmarks via attention scoring
     - Captura un "resumen global" del pasado completo
     - Costo: 1 forward pass de un Linear(d_model, 1)

5.3 SOLUCIÓN: NSA V3 CON TRIPLE ATENCIÓN (nsa_module_v3.py)

  El módulo NSA ahora tiene TRES tipos de atención:
  
  1. Causal Local Attention (FlashAttention/SDPA)
     - Atiende a los tokens en la ventana cercanea
     - Precisión perfecta para contexto reciente
     
  2. Current Landmark Cross-Attention
     - Atiende a landmarks del chunk actual (mean-pooled de Path 1)
     - Conecta lo local con lo global del chunk actual
     
  3. ARCHIVED Historical Cross-Attention ← NUEVO
     - Atiende a embeddings de landmarks congelados del pasado lejano
     - KV projection separada (archive_kv_proj) para no contaminar el espacio local
     - Top-4 landmarks más recientes + 1 global summary
     - Costo: O(S × 5 × d) — constante respecto a longitud total
  
  Gated Combination:
     - out = W_combine · [out_local ‖ out_landmark ‖ out_archive]
     - El modelo APRENDE cuánto peso dar a cada fuente

5.4 SOLUCIÓN: EMA ADAPTATIVO (sdpc_engine_v3.py)

  Problema: Con momentum fijo (0.9), la EMA se satura en secuencias muy largas
  Solución: Decay progresivo del momentum
  
  - En 0 tokens: μ = 0.9 (fuerte memoria, acumula información)
  - En 10M tokens: μ = 0.7 (más reactivo a info nueva)
  - Fórmula: μ = 0.9 - min(tok/10M, 1) × 0.2
  
  Esto previene que el estado EMA se "congele" y deje de reaccionar.

5.5 SOLUCIÓN: MULTI-HEAD EN LUGAR DE ALTO GRADO

  Problema: Subir de grado 4 a grado 12 causa:
  - 3× más operaciones por token (latencia lineal)
  - Presión de registros: 12 coeficientes c₀-c₁₁ saturan registros GPU
  - Fenómeno de Runge: oscilaciones salvajes amplificadas exponencialmente
  
  Solución adoptada: 4 cabezas × grado 4 = "16 dimensiones de resolución"
  - Matemáticamente más estable que 1 cabeza × grado 16
  - Cada cabeza se especializa en diferentes frecuencias temporales
  - GPU paraleliza las 4 cabezas (pid_h como dimensión de grid)
  - No satura registros: cada thread maneja solo 4 coeficientes

§9. VERSIÓN 4 — PARALLEL SCAN + DEGREE 8 + INTELLIGENT LANDMARKS
================================================================================

9.1 PARALLEL SCAN (Supera a Mamba en su propio juego)

  ARQUITECTURA DEL KERNEL REESCRITA COMPLETAMENTE.
  Approach anterior (V1-V3): bucle secuencial for s in range(seq_len)
  Approach V4: Three-phase parallel

  FASE 1 — Parallel Clenshaw (Triton kernel)
    - UN programa por posición secuencial (embarrassingly parallel)
    - Grid: (B × n_heads, seq_len, cdiv(head_dim, BLOCK_HD))
    - Los coeficientes son FROZEN para toda la secuencia → sin dependencia
    - Todas las posiciones se procesan simultáneamente
    - Incluye SiLU post-Clenshaw + clamping anti-Runge ±2.0

  FASE 2 — Associative EMA Scan (PyTorch)
    - Construye una matriz causal [C×C]: M[i,j] = (1-μ)·μ^(i-j)
    - Aplica via torch.einsum('tj,bcjd->bctd') — paralelismo completo
    - Carry correction: prefix scan secuencial solo en n_chunks (32 iters para 8K)
    - Operator asociativo: (a₁,b₁) ⊕ (a₂,b₂) = (a₁·a₂, a₂·b₁+b₂)

  FASE 3 — Parallel TTT + Adam (PyTorch)
    - Chebyshev basis T_k(x) computada en paralelo para todas las posiciones
    - Gradiente: mean(err · T_k) / degree sobre toda la secuencia
    - Adam update con lr diversity factor 0.35 por cabeza
    - RMSNorm soft bounding sobre coeficientes

  RESULTADO: 10M tokens con 0.00% state drift, 1.47M tok/s

9.2 DEGREE 8

  - 8 coeficientes c₀-c₇ por cabeza (antes 4)
  - Clenshaw extendido: u₈=0, u₇=c₇, ..., y=x·u₁-u₂+c₀
  - Output clamping ±2.0 para prevenir Runge phenomenon
  - TTT gradient escalado por 1/degree para estabilidad

9.3 8 CHEBYSHEV HEADS

  - 8 cabezas × head_dim=32 (con d_model=256)
  - 64 dimensiones espectrales totales (8 heads × 8 degree)
  - Cada cabeza tiene lr adaptativo:
    lr_h = base_lr × (1 + 0.35 × (h - H/2) / H)
  - Cabeza 0: lr = 0.0083 (retenedora, memoria larga)
  - Cabeza 7: lr = 0.0118 (reactiva, corto plazo)

9.4 LANDMARK IMPROVEMENTS

  - max_landmarks=64 (antes 32), k=8 (antes 4)
  - Weighted merge: 65% peso al landmark más reciente, 35% al más viejo
  - Self-attention entre landmarks (nn.MultiheadAttention, 4 heads)
  - Global summary: attention-weighted average de todos los landmarks

9.5 STATE REFRESH FROM LANDMARKS

  - Cada refresh_interval=32768 tokens:
    1. Toma el summary de los landmark embeddings
    2. Lo proyecta a través de un MLP (d→d/2→n_heads×head_dim)
    3. Gate sigmoid: controla cuánta info se inyecta
    4. Suma gated al estado de coeficientes: c += 0.1 × gate × refresh
  - Esto le da al estado recurrente "memoria semántica" del pasado

9.6 3-LEVEL DYNAMIC DEGREE
  
  - complexity < 0.2 → degree 2 (rápido, simple)
  - complexity < 0.5 → degree 4 (medio)
  - complexity ≥ 0.5 → degree 8 (máxima resolución)

11.1 PROBLEMA IDENTIFICADO — 4 COMPONENTES CON DEAD GRADIENT
--------------------------------------------------------------
  El benchmark T7 del benchmark_chimera_deep.py reveló que 4 de las rutas
  críticas de CHIMERA tenían gradiente nulo, haciendo que sus parámetros
  fueran efectivamente no entrenables:

  (a) ttt_U, ttt_V, ttt_full_scale → GRAD=0
      Causa: hard threshold `if probs[:,2].mean() > 0.35` — el router
      inicializa uniforme (prob_full ≈ 0.333 < 0.35) → rama TTT-Full
      nunca se ejecuta → W_U, W_V, scale nunca reciben gradiente.
      BUG ADICIONAL: ambas matrices inicializadas con torch.zeros(). Incluso
      al eliminar el threshold, con U=0 y V=0:
        h = mamba_out @ V.T = 0
        ∂L/∂U = h^T @ grad_corr = 0^T @ grad = 0  (punto de silla exacto)

  (b) slr.lam_logit → GRAD=0
      Causa: lam_logit usado con .item() → escalar Python → se desconecta
      del grafo de autograd → no recibe gradiente.

  (c) bus.gate, bus.publish → GRAD=0 / GRAD=None
      Causa: `if bus_cache is None: return x, summary_unsqueezed` →
      cuando el cache está vacío (primera capa o init), el gate y el
      path de modulación nunca se ejecutan → sin gradiente.

11.2 FIXES APLICADOS
--------------------
  FIX 1 (advanced_chimera.py — TTT-Full dead gradient):
    ANTES: `if probs[:,2].mean() > 0.35: scale = sigmoid(ttt_full_scale)...`
    AHORA: Siempre activo, escalado por prob_full como soft-gate:
      scale = sigmoid(ttt_full_scale)
      correction = mamba_out @ V.T @ U.T
      mamba_out += prob_full * scale * correction
    → prob_full ≈ 0.31 en init → contribución pequeña pero gradiente SIEMPRE fluye

  FIX 2 (advanced_chimera.py — ttt_U/V initialization):
    ANTES: zeros(d_model, rank), zeros(rank, d_model)  → punto de silla
    AHORA: randn * (0.01/sqrt(rank)) para ambos → correction ≈ O(1e-5),
    warm-up suave pero con pendiente no-nula desde el paso 0.

  FIX 3 (advanced_chimera.py — bus.gate early return):
    ANTES: `if bus_cache is None: return x, summary_unsqueezed`
    AHORA: trivial_mod = modulate(summary.expand(-1,S,-1)) * sigmoid(gate)
           return x + trivial_mod, summary_unsqueezed
    → gate y modulate reciben gradiente incluso en primera capa.

  FIX 4 (sgr_slr.py — SLR V3 reescritura):
    - DiffAttnV2Function(autograd.Function): backward analítico correcto
      ∂L/∂Q1, ∂L/∂K1, ∂L/∂V, ∂L/∂lam_logit (como tensor, no .item())
    - FusedProjectionSplit: W_q [D,2*dh] + W_kv [D,3*dh] → 2 cuBLAS vs 5
    - _gather_windows_batched(): torch.gather batched, zero .item()
    - scatter_add_ sobre zeros_like → elimina clone() del tensor [B,S,D]
    - @triton.autotune (6 configs) en kernel de inferencia

  FIX 5 (ttt_kernel.py — 2D grid para compute_token_errors):
    ANTES: grid=(Sm1,) con for b in range(B) Python → B lanzamientos serializados
    AHORA: grid=(B, Sm1) → un solo dispatch para todo el batch
    → B veces menos overhead de lanzamiento de Triton

  FIX 6 (ttt_kernel.py — zero-copy Lion update):
    ANTES: siempre .float().contiguous() → 3 copias intermedias + copy_back
    AHORA: if tensor.dtype==float32 and tensor.is_contiguous(): usa directo
    → zero-copy cuando dt_bias/momentum ya son fp32 (caso normal en Mamba2)
