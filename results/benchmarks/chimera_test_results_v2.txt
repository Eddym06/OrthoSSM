  5. ANÁLISIS DE PROBLEMAS CONOCIDOS
────────────────────────────────────────────────────────────────────────

  [A] Drift acumulado en chunked forward (T5 deep_analysis):
    Síntoma: max_err=3.46, drift=+108.4% (full vs chunked a S=2048)
    Causa: discretización de dt_bias entre chunks (estado SSM no se lleva)
    Impacto: training — los gradientes pueden ser ligeramente incorrectos
    en secuencias largas procesadas en chunks
    Severidad: BAJA para entrenamiento normal (<4K tokens por chunk)
    Acción (si se necesita): implementar "chunked carry" donde el estado
    SSM (h, conv_state) se pasa entre chunks. Ver backlog [D3/D5].

  [B] Gradientes finite-difference (T6 deep_analysis):
    Los rel_err son ALTOS pero esperados: mamba2.dt_bias=2.15, router=0.45
    Causa: gradientes a través de discretización exp() son sensibles a h
    Impacto: NO es problema — SGD/Adam convergen igual, rel_err FD no
    necesita ser <0.1 para modelos de alto grado
    Acción: monitorear NaN/Inf en training; no se recomienda cambiar nada

  [C] Token #1 lento (12ms) en bench_decode_stability:
    Causa: Triton JIT compila el kernel en el primer call
    Impacto: solo en el primer token de la primera sesión
    Acción: implementar warmup de 3-5 tokens antes de medir latencia
    (make_cuda_graph_step ya incluye warmup_iters=3 por defecto)

  10.2 CUDAGraphPool (batch_size pool para serving real)
────────────────────────────────────────────────────────────────────────

  PROBLEMA: make_cuda_graph_step() capturaba UN único batch_size.
  En serving real el número de requests varía (1, 2, 4, 8 por batch).
  Si llega B=3 al sistema, el único graph de B=1 es inútil → fallback CPU.

  IMPLEMENTACIÓN:
    class CUDAGraphPool:
      - Pre-captura un CUDA Graph por cada batch_size en la lista
      - _select_batch_size(B): padding al mayor disponible ≥ B
        (B=3 → usa graph B=4, filas 3: rellenas con ceros)
      - step(x, cache): selecciona graph automáticamente por x.shape[0]
      - allocate_cache(B): reserva estados compatibles para batch_size B

  BATCH_SIZES ADAPTATIVOS POR VRAM:
    RTX 4050 (6.4 GB VRAM):  [1, 2, 4]         (3 graphs pre-capturados)
    GPUs ≥12 GB:              [1, 2, 4, 8]      (4 graphs)
    H200 (≥40 GB VRAM):       [1, 2, 4, 8, 16]  (5 graphs)

  RESULTADO EN RTX 4050:
    Graphs capturados: [1, 2, 4]
    B=1 out: [1, 1, 256] ✓ | B=2: [2, 1, 256] ✓ | B=3→pad4: [3, 1, 256] ✓
    Latencia idéntica a single-graph (VRAM fix, no perf regression)

  10.3 Fp8Linear + apply_fp8_to_chimera (FP8 forward Hopper/Blackwell)
────────────────────────────────────────────────────────────────────────

  IMPLEMENTACIÓN:
    class Fp8Linear(nn.Module):
      - Cuantificación dinámica por tensor: scale = amax(|x|) / 448.0
      - torch._scaled_mm(x_fp8, w_fp8.T, scale_x, scale_w) → BF16 output
      - Backward en BF16/FP32 (FP8 solo en forward pass)
      - try/except fallback a BF16 si _scaled_mm falla (runtime error)
      - from_linear(nn.Linear) → copia pesos sin reentrenamiento

    def apply_fp8_to_chimera(layer, prof=None):
      - Convierte: bus.publish, bus.gather_q, bus.modulate, slr.out_proj
      - NO convierte: mamba2 internals (librería externa), router.mlp
      - Condición de activación: prof.use_fp8_fwd = True (SÓLO Hopper/Blackwell)
      - En RTX 4050/A100: NO-OP (use_fp8_fwd=False en gpu_profile.py)

  DISPONIBILIDAD FP8 EN HARDWARE:
    RTX 4050 (SM=8.9):   _FP8_AVAIL=True, _scaled_mm=True
                         PERO use_fp8_fwd=False (BW-limited, no compute-limited)
    H100/H200 (SM=9.0):  use_fp8_fwd=True → activo
    B200   (SM=10.0):    use_fp8_fwd=True → activo

  MEDICIÓN EN RTX 4050 (con _scaled_mm disponible):
    Fp8Linear rel_err vs nn.Linear: 0.0378 (3.78%) — dentro de tolerancia BF16
    Shape [4, 256]: ✓ | NaN/Inf: ninguno
    apply_fp8_to_chimera: no-op correcto en laptop_ada ✓

  SPEEDUP ESPERADO EN H100/H200:
    FP8 vs BF16 throughput (NVIDIA spec): 2× para matmul compute-limited
    Lineales afectadas: ~35% del tiempo total de forward (T4: bus+SLR ~56%)
    Speedup neto estimado: 1.3-1.7× en layers compute-limited (H100/H200)
    RTX 4050: no aplica (bandwidth-limited, no compute-limited)

  10.4 chunked_prefill (prefill O(chunk_size) VRAM para S > 4K)
────────────────────────────────────────────────────────────────────────

  PROBLEMA ORIGINAL (T5 drift = +108%):
    Llamadas separadas a layer.forward(chunk_i, bus_cache=None) reiniciaban:
    a) El ring buffer del BUS → tokens subsiguientes sin contexto previo
    b) El estado SSM interno de Mamba2 (conv_state + ssm_state) → cada chunk
       veía una secuencia "fresca" sin memoria de chunks anteriores

  QUÉ RESUELVE chunked_prefill (implementación final):
    AMBAS fuentes de drift resueltas:

    1. BUS CARRY: bus_cache se lleva entre chunks → cada chunk ve resúmenes de
       todos los tokens anteriores vía ring buffer. Elimina drift de BUS.

    2. SSM CARRY (via Mamba2ChunkedPrefill + _mamba2_carry_ctx):
       causal_conv1d_fn(initial_states=conv_state, return_final_states=True)
         → propaga conv_state entre chunks
       mamba_chunk_scan_combined(initial_states=ssm_state, return_final_states=True)
         → propaga ssm_state entre chunks
       _mamba2_carry_ctx(layer, B) parchea layer.mamba2.forward temporalmente
       con una versión stateful (Mamba2ChunkedPrefill.forward_chunk) mientras dura
       el contexto del for-loop de chunks, luego restaura el forward original.
       Requiere: mamba_ssm≥2.0, causal_conv1d≥1.4 (initial_states API).

  VALOR PRINCIPAL (VRAM):
    Sin chunking: O(S) activaciones  → 1M tokens = 1 GB VRAM (imposible en 6GB)
    Con chunking: O(chunk_size) fijo → 4K chunk = 4.2 MB (constante ✓)

  CHUNK_SIZE DEFAULTS POR GPU:
    RTX 4050:   2048 tokens/chunk (6.4GB VRAM)
    A100 80GB:  8192 tokens/chunk
    H200 140GB: 16384 tokens/chunk

  IMPLEMENTACIÓN:
    def chunked_prefill(layer, x, chunk_size=None, bus_cache=None, return_aux=False):
      if S <= chunk_size: return layer(x, bus_cache=bus_cache)   # bypass
      with _mamba2_carry_ctx(layer, B):          # parchea mamba2 con carry SSM
          for chunk in split(x, chunk_size):
              out, bus_cache = layer(chunk, bus_cache=bus_cache)  # carry bus_cache
      return cat(outputs), bus_cache
